### OS-ADL-023: SQLite-Native Retrieval Engine
**Status:** LOCKED | **Date:** 2026-04-04 | **Domain:** architecture | **Supersedes:** OS-ADL-004
**Decision:** Retrieval built natively on SQLite using three complementary search modes fused via Reciprocal Rank Fusion (RRF): (1) **sqlite-vec** for vector similarity search (cosine distance on 384-dim embeddings from fastembed/bge-small-en-v1.5, registered as C extension via `sqlite3_auto_extension`), (2) **FTS5** for keyword/BM25 search (already in project since V7), (3) **entity/relationship tables** with recursive CTEs for graph traversal (temporal edges via `valid_from`/`valid_until`). `RetrievalBackend` trait defines the interface — `SqliteRetrievalBackend` is the shipped implementation; trait boundary enables future `LanceRetrievalBackend` if project scale exceeds ~100K vectors. Three-tier mana cost: sigil metadata queries (0), hybrid FTS5+vector search (1), deep graph traversal with LLM relevance scoring (3). Metadata pre-filtering via SQL WHERE on entity tags/domain/persona before vector math — structured vault metadata eliminates candidates before expensive operations. Incremental indexing tracks content hashes, re-embeds only changed files.
**Rationale:** (1) SQLite already in project at V15 — sqlite-vec adds one C extension, not a new database. (2) Single database = atomic transactions across retrieval + application data (find a trace, update a signal, file a finding — one commit). (3) Vault content is structured (YAML frontmatter, tags, typed traces) — SQL exploits this structure; graph RAG ignores it. (4) At vault scale (<50K vectors), brute-force cosine similarity is <100ms — no ANN index needed. (5) No Python, no external processes, no extra install requirements — ships in the Tauri binary. (6) LightRAG benchmarks inflated by evaluation bias; LazyGraphRAG outperformed in all 96 tests; CVE-2026-30762 disclosed day of decision. (7) RetrievalBackend trait preserves upgrade path to LanceDB for the vector layer if a project's combined vault + repo exceeds ~100K vectors — FTS5 and graph stay in SQLite regardless.
**Consequence:** `src-tauri/src/retrieval/` module: `backend.rs` (trait), `sqlite_backend.rs` (impl), `embedder.rs` (fastembed wrapper), `entities.rs` (entity store), `relationships.rs` (relationship store + temporal edges), `leylines.rs` (bidirectional backlink generation). SQLite V16 amendment adds `vec_embeddings` virtual table, `entities` table, `relationships` table, `content_hashes` table. ~12 new Tauri commands. Rust dependencies: `fastembed` (ONNX Runtime, ~67MB model download on first use), `sqlite-vec` (C extension). No Python. No external database process.

### OS-ADL-023b: Virtual Filesystem over Vault
**Status:** LOCKED | **Date:** 2026-04-04 | **Domain:** architecture
**Decision:** Vault exposes itself as a virtual filesystem backed by SQLite — agents navigate via familiar operations (ls/cat/grep analogues) instead of raw file reads. Path tree derived from sigil indexes (zero-cost navigation). Per-persona vault tree pruning via RBAC: each persona sees only content in their domain (Tanaka sees auth + security, Mara sees frontend + UX, pruned content is absent, not hidden). Composable filesystem mounts: vault components (skills, ADL, echoes, sigils) independently mountable per dispatch — read-only gate dispatch mounts skills + ADL, dreamtime mounts everything. Lazy content resolution: large artifacts stored as pointer entries, materialized on explicit read at 1 mana cost.
**Rationale:** Raw file reads give agents unconstrained access to the entire vault — wasteful context and potential information leakage between domains. Per-persona pruning ensures agents only see content relevant to their expertise. Composable mounts align with capability grants from Phase 7 — the filesystem respects the permission model. Lazy pointers prevent large artifacts from bloating context windows.
**Consequence:** `src-tauri/src/vault/virtualfs.rs`, `pruning.rs`, `mounts.rs`. 5 Tauri commands (vault_ls, vault_read, vault_search, get_persona_tree, mount_components). Pruning rules derived from persona `CapabilityFamily` assignments. Mount composition configured per dispatch via capability grants.

### OS-ADL-023c: Knowledge Garden Visualization
**Status:** LOCKED | **Date:** 2026-04-04 | **Domain:** visualization
**Decision:** The Graph Viewer panel is upgraded from Canvas 2D force-directed layout (Phase 5.3) to a react-three-fiber Knowledge Garden with four tabs. Visual reference: poetengineer (Kat Zhang) Session Garden / Idea Garden — L-system botanical trees growing from void. Each domain cluster is a tree: trunk is the foundational entity, branches fork at relationship points, colored dots at tips are terminal nodes. Color-coded by domain tag using existing PERSONA_COLORS + distinct content-domain palette. Default top-down view (observatory register — dot clusters against dark void). `OrbitControls` enables 2D→3D perspective tilt revealing tree depth. Timeline scrubber replays garden growth chronologically via temporal entity queries. Cross-pollination particles animate along cross-domain ley line edges. Selective bloom (`UnrealBloomPass`) — active/hovered nodes glow, others stay matte. Tag pill sidebars as legend + filter. Four tabs: Grimoire (standard node-link for agent registry), Vault/Garden (L-system trees by domain), Ley Lines (garden + cross-pollination particles), Scrying (orbital layout for intelligence chains with Arbiter center). Observatory aesthetic: luminous data against restrained void, energy from particle motion and selective glow, not saturated color.
**Rationale:** The garden metaphor maps directly to the vault's temporal, relational nature — trees grow as knowledge accumulates, branches show how ideas fork, cross-pollination shows how domains influence each other. The 2D→3D perspective shift gives two registers of attention: observatory (read the forest) and garden (read a single tree). react-three-fiber provides WebGL rendering with React component model — bloom, particles, instanced meshes for performance, raycasting for interaction. The existing Canvas 2D renderer (graph-layout.ts) was adequate for placeholder data but cannot deliver bloom, particles, or 3D perspective.
**Consequence:** Frontend dependencies: `@react-three/fiber`, `@react-three/drei`, `@react-three/postprocessing`, `three`. GraphViewerPanel.tsx major rewrite. New files: `garden-layout.ts` (L-system tree + forest positioning algorithm), `garden-scene.tsx` (Three.js scene component), `garden-detail.tsx` (DOM overlay detail panel), `tag-sidebar.tsx` (filterable tag pills). `useGraphData.ts` rewritten to query retrieval engine via Tauri bridge. Reference screenshots: `observatory-reference/` (30 images from poetengineer).
